## [PR#3258](https://github.com/openhwgroup/cva6/pull/3258) Add SvpbmtEn flag and pbmt struct fields 
## Summary

Adds scaffolding for Svpbmt (page-based memory types), mandatory
in RVA23 and currently completely absent from CVA6. Five inert
structural changes with zero functional impact when `SvpbmtEn=0`
(the default for all existing configs).

Part of [#3253](Issue3253.md)

This PR introduces the initial scaffolding only (5 files, no functional change when SvpbmtEn=0).

Follow-up work will extend this with PTW assignment, TLB storage, CSR wiring, LSU integration, and cache backend support.
## Changes

| File | Change |
|------|--------|
| `include/config_pkg.sv` | Add `bit SvpbmtEn` after `SvnapotEn` in both `cva6_user_cfg_t` and `cva6_cfg_t` |
| `include/build_config_pkg.sv` | Propagate `SvpbmtEn` into built config (mirrors `SvnapotEn` pattern) |
| `core/cva6_mmu/cva6_mmu.sv` | Split `pte_cva6_t.reserved[8:0]` → `pbmt[1:0]` + `reserved[6:0]` per Svpbmt spec |
| `core/cva6_mmu/cva6_mmu.sv` | Add `logic [1:0] pbmt` to `tlb_update_cva6_t` after `is_napot_64k` |
| `core/cva6_mmu/cva6_ptw.sv` | Update comment at invalid-PTE check: `pte.reserved` now covers bits [60:54] only |

## Template

`SvpbmtEn` follows the `SvnapotEn` pattern exactly — optional,
parameter-gated, disabled by default (parachute rule satisfied).
All future functional code will be gated on `CVA6Cfg.SvpbmtEn`.

`pte_cva6_t.reserved[8:0]` currently buries PTE bits [62:61]
(the MT field) inside the reserved check at `cva6_ptw.sv:430`,
causing a page fault on any Svpbmt-typed page. After this split,
`pte.reserved` is 7 bits so `|pte.reserved` naturally excludes
the MT field — no logic change required.

`tlb_update_cva6_t` is a `localparam type` defined only at
`cva6_mmu.sv:124` and passed as `parameter type` to all three
submodules — one field addition propagates everywhere
automatically, the same mechanism used for `is_napot_64k`.

## Testing

With `SvpbmtEn=0` (all existing configs): no functional change.
CI should pass unmodified.
